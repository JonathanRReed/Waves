## How it was done

Each phase ran as a multi-agent workflow; every code change was gated by `swift build` + `swift test`.

1. **Inventory** — 30 feature-area readers + a completeness critic wrote one user story per feature with expected behavior, acceptance criteria, code refs, and an honest implementation status.
2. **Test** — 31 testers + 31 adversarial verifiers checked every story's acceptance criteria against the real code and documented every defect; 9 candidate findings were rejected as false positives.
3. **Triage + Fix** — defects were clustered into root-cause fix units (fix-now vs defer), then applied in severity tiers (critical+high → medium → low → gap-fill), AppStore.swift serialized to avoid conflicts, the rest parallel by file.
4. **Re-test** — every story re-verified against the modified code + adversarial diff review of the most-changed files. 31 follow-ups (incomplete fixes / regressions) were found, remediated, and a final 8-file verification confirmed 28/28 resolved with one regression — which was then fixed and re-verified.

## Highlights of what was fixed

- **Excluded-app safety (critical):** excluded apps could be re-tapped via the volume hotkeys, the set-volume URL scheme, and per-device-preset restore on device change — all now guarded.
- **Pinned per-app output route (critical):** on relaunch a pinned app silently played to the system default while the UI claimed it was managed — relaunch now actually re-applies the saved device route.
- **Route-health honesty:** failed volume changes no longer silently wipe Error chips; a plain refresh no longer flips health green without a real re-apply; the global badge no longer reads "Ready" while apps are in Error.
- **Device-change UI:** the picker/checkmark no longer go stale when "auto-restore" is off; per-device preset restore is decoupled from that toggle; manual switches no longer double-toast.
- **Import/export:** preset import is now atomic (no partial apply of an invalid backup), uses a versioned envelope, gives clear errors, and the panels anchor to the hosting window.
- **Accessibility:** combined labels on loading spinners, non-color status cues on diagnostics, scoped "needs attention" rotor.
- Plus dead-code removal, value clamping, localized sorting, toast timing/eviction, and onboarding accuracy fixes.

## Deferred backlog (102 items — see `Fix Status`/`Errors Found` columns in the CSV)

Severity of deferrals: **16 high · 40 medium · 46 low.** They cluster into themes worth prioritizing:

- **Live-metering reactivity (high/med):** the menu-bar icon glyph, "Playing audio" labels, and Activity sort read stale session levels rather than the live meter stream (`MENU-01`, `LVL-04`, `SORT-02`, `ROUTE-04`). A reactive wiring change — needs design + on-device verification.
- **Auto-pause music (high/med):** browser-based calls (Meet/Zoom-web/Teams-web) aren't detected; pause triggers on window focus rather than an actual live call; resume can fire mid-call (`PAUSE-01/02/03`). Needs real call/mic-activity detection — a feature.
- **"Auto-restore device" toggle is a no-op (high):** route re-establishment happens inside the backend's CoreAudio listener (`WorkspaceAudioControlBackend.swift:1571`), which can't see the preference. Making the toggle functional needs the preference threaded into the backend **and** a decision on off-semantics (dispose-but-don't-reattach) — requires testing on real device switches (`ROUTE-05`, `DEV-04`, `OUT-06`).
- **Onboarding accuracy (high/med):** the checklist can show a false "You're all set!" from a stale persisted status; step completion keys off OS-version capability alone; "Test Recovery" shows success even when routes stay unhealthy (`ONB-01/02/04`, `SETUP-04`). Product decisions about checklist semantics.
- **Hotkey feedback while backgrounded (high/med):** global hotkeys give no visible confirmation when Waves isn't frontmost (the toast renders in-window only) and are silent when Accessibility is denied (`KEY-01/02/03/05`). Needs an OS-level HUD/notification surface — a new feature.
- **Relaunch cache reconciliation (high):** apps not running at relaunch lose saved boost/state because the snapshot is assigned without merging cached-only apps (`BOOST-04`).
- **Menu-bar parity & misc (med):** no per-app meters in the menu panel, `showMenuBarExtra` has no UI toggle, "Recent" scope is a misnomer, per-device volume-control mode is written-but-never-read (`LVL-01`, `MENU-06`, `DISC-08`, `AUDIO-01`).

These were deferred deliberately — they need either your product judgment or a Mac to validate audio-routing behavior, not because they're unclear. Each row in the spreadsheet carries the full detail, code reference, and suggested fix.

## Decisions to unblock the backlog

A second automated pass fixed 9 more safe items (false-success recovery toasts, dead-code removal, a login-item persistence fix, an empty-device-list row, a help tooltip) and confirmed the remaining 93 are genuinely blocked on a decision or hardware — not on effort. Here's what each cluster needs from you so the next round can implement it. **Recommendations are mine; pick or override, then re-run `/loop` scoped to a theme.**

| Theme | What's blocked | Decision needed | My recommendation |
|---|---|---|---|
| **Live-metering reactivity** (`MENU-01`, `LVL-04`, `SORT-02`, `ROUTE-04`) | Menu-bar glyph, "Playing audio" labels, and Activity sort read static snapshot levels, not the live poller | Should the live level stream drive icon/labels/sort, accepting periodic re-sort cost? | **Yes** — drive icon + "playing" state from the existing `liveLevels` poller; keep Activity sort on a throttled (≥1s) tier so it doesn't re-sort every tick. Needs on-device check. |
| **Auto-pause music** (`PAUSE-01/02/03`, `MUTE-03`) | Only frontmost-focus triggers pause; browser calls undetected; resume fires mid-call | Detect a real call via mic/input-in-use, or keep focus-based? | Add a mic/input-in-use probe (`kAudioDevicePropertyDeviceIsRunningSomewhere`) to gate pause/resume — a real feature; validate on hardware. If you'd rather not, I can at least make resume require the app to *quit*, not just lose focus. |
| **"Auto-restore device" toggle** (`ROUTE-05`, `DEV-04`, `OUT-06`) | Toggle is a no-op; routes always re-establish in the backend's CoreAudio listener | Thread the preference into the backend, and what should "off" do? | Thread `autoRestoreDevice` through the backend protocol; on **off**, *dispose* managed routes on device change but **don't** reattach (apps follow system default). Must be tested on real device switches. |
| **Onboarding accuracy** (`ONB-01/02`, `SETUP-01/03`) | False "You're all set!" from stale status; step completes on OS-version alone; capture permission not surfaced as a step | Restructure the checklist: add a capture-permission step and require live (not persisted) readiness? | **Yes** — add an explicit "Audio capture permission" step with a Grant action, and seed flags from a live probe, not persisted `backendStatus`. |
| **Hotkey feedback when backgrounded** (`KEY-01/02/03/05`) | Global hotkeys give no visible confirmation when Waves isn't frontmost; silent when Accessibility denied | Add an OS-level HUD/notification surface? | Add a lightweight borderless floating HUD (like the system volume HUD) for hotkey actions; show a one-time prompt when Accessibility is denied. A feature. |
| **Relaunch cache reconciliation** (`BOOST-04`) | Apps not running at relaunch lose saved boost/state because the snapshot is assigned without merging cached-only apps | What's the retention/expiry policy for not-running apps in the mixer? | Keep cached-only apps with saved settings for N days (e.g. 14) via `cleanupStaleEntries`; merge them into the snapshot on refresh so settings survive a relaunch. |
| **Menu-bar parity & reachability** (`LVL-01`, `MENU-06`, `PIN-04`, `DISC-08`, `AUDIO-01`) | No per-app meters in the menu panel; `showMenuBarExtra` & pin-from-menu have no UI; "Recent" is a misnomer; per-device volume-control mode is written-but-never-read | Several small UX adds — want them? | Add menu-panel per-app meters + a pin control + a `showMenuBarExtra` toggle in Settings; rename "Recent" → "Other"; either wire or remove the dead per-device volume-control mode. Low-risk; mostly UI. |

Everything else in the 93 is a smaller variant of one of the above or a single-row UX/inconsistency call — all itemized in the CSV with severity, code ref, and a suggested fix.
