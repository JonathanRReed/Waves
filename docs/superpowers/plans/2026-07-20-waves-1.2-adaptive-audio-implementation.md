# Waves 1.2 Adaptive Audio Implementation Plan

## Phase 1: Baseline and worktree reconciliation

1. Inspect every existing uncommitted change and assign it to icon, onboarding,
   persistence hardening, release metadata, or unrelated work.
2. Run strict formatting, `swift build`, and `swift test` without modifying the
   current artifact set.
3. Preserve all valid existing changes and repair any baseline failure before
   adding 1.2.0 features.

## Phase 2: Audio models and DSP

1. Add Managed Audio EQ settings and persistence.
2. Add content type, priority, and strategy models with backward migration.
3. Extend backend contracts for global EQ propagation.
4. Add the second EQ layer and combined headroom handling to tap controllers.
5. Replace role-only adaptive math with the priority matrix and content-aware
   activity rules.
6. Add focused model, DSP, migration, and backend tests.

## Phase 3: Appearance and visual shell

1. Add appearance and palette preferences.
2. Remove the unconditional dark appearance override.
3. Make the Waves and Graphite palettes adaptive across light and dark modes.
4. Apply one coherent shell to main mixer, inspector, settings, menu bar, and
   onboarding.
5. Clean up selected rows, headers, list spacing, seams, empty states, and
   inspector hierarchy.
6. Add appearance controls to Settings.

## Phase 4: Sound workspace

1. Add the Sound navigation destination.
2. Build Managed Audio EQ controls with presets, band modes, and headroom state.
3. Build Adaptive Mix strategy and active-app policy controls.
4. Refine the per-app inspector around EQ and Adaptive sections.
5. Add accessibility labels, help text, keyboard navigation, and disabled-state
   explanations.

## Phase 5: Onboarding and repair

1. Centralize System Settings destinations and repair actions.
2. Build the four-stage first-run flow.
3. Add basic appearance, menu-bar, launch, recent-app, shortcut, and adaptive
   preferences to Personalize Waves.
4. Build Setup & Repair with contextual actions and automatic re-checking.
5. Add non-destructive Redo Setup entry points from Settings, diagnostics, and
   Help.
6. Add onboarding and repair tests.

## Phase 6: Completion verification

1. Run strict formatting, build, and all tests.
2. Run deterministic multi-stream adaptive and EQ scenarios.
3. Build and launch the fresh app bundle.
4. Inspect every requested UI state in both palettes and appearances.
5. Verify permission deep links and repair re-check behavior in System Settings.
6. Verify menu-bar, Settings, onboarding, and main-window consistency.
7. Run the packaged-app verification skill and release scripts.

## Phase 7: Release 1.2.0

1. Align changelog, app version, build number, cask, and appcast inputs.
2. Commit the complete verified change set with conventional commits.
3. Push the exact release commit to `origin/main`.
4. Build, sign, notarize, staple, and validate the universal DMG.
5. Publish the GitHub release and tag from the exact main commit.
6. Generate, validate, and deploy the Sparkle appcast and site metadata.
7. Verify the Sparkle update from installed 1.1.0 to 1.2.0.
8. Verify the final `/Applications/Waves.app` bundle, version, signature,
   notarization, icon, UI, and runtime behavior.
