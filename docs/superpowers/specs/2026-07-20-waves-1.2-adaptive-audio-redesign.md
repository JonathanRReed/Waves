# Waves 1.2 Adaptive Audio Redesign

## Status

Approved for implementation on 2026-07-20.

## Objective

Turn Waves into a cohesive, theme-aware macOS audio utility with:

- a cleaner and more consistent main window;
- user-selectable light and dark appearances;
- a dedicated managed-audio equalizer;
- persistent per-app content types and priorities;
- adaptive behavior that supports lecture, meeting, music, and media scenarios;
- a guided first-run setup and non-destructive repair flow;
- a signed, notarized, published 1.2.0 update.

The implementation must preserve Waves' native, quiet, and precise product
character. It must not become a DAW, plugin suite, or decorative dashboard.

## Product Boundaries

### Managed audio, not every sound on macOS

Waves processes apps that are routed through its Core Audio process-tap path.
The shared equalizer is therefore named **Managed Audio EQ**. It applies to all
managed app streams and does not claim to affect excluded, unsupported, or
otherwise unmanaged system audio.

### Existing audio architecture remains authoritative

Waves keeps the current per-app tap-controller architecture. A true system-wide
output bus would add device, latency, feedback, and route-recovery risks that are
not justified for this release.

### Permission repair stays user-controlled

Waves can detect missing permissions, open the correct System Settings pane,
explain the exact action, and automatically re-check when the app becomes active.
It cannot toggle protected macOS permissions on the user's behalf.

## Experience Design

### Appearance and themes

Waves provides two curated palettes, each with light and dark rendering:

1. **Waves**: navy surfaces and signal cyan, preserving the current identity.
2. **Graphite**: neutral graphite or cool-gray surfaces with a restrained teal
   signal accent.

Appearance is selected independently:

- **System** follows the current macOS appearance;
- **Light** forces the selected palette's light variant;
- **Dark** forces the selected palette's dark variant.

The app keeps system fonts, native controls, semantic status colors, Increase
Contrast support, and Reduce Transparency support. Accent color remains rare and
is reserved for live signal, current selection, and primary actions.

### Main window cleanup

The main window uses one coherent themed shell rather than mixing unrelated
`windowBackgroundColor`, `textBackgroundColor`, `.bar`, and branded surfaces.

The cleanup includes:

- a single adaptive background system for the main mixer and inspector;
- consistent header, list, selected-row, and inspector surfaces;
- a quieter selected-row treatment that does not create a large gray slab;
- stable top alignment and spacing when only one or two live apps are visible;
- a refined inspector with clear EQ and Adaptive sections;
- removal of unnecessary nested cards and hard seams;
- readable light-mode contrast for all labels, meters, controls, and states;
- responsive behavior across the supported main-window widths;
- preserved keyboard, VoiceOver, Reduce Motion, and Increase Contrast behavior.

### Dedicated Sound workspace

A new **Sound** destination appears in the main sidebar. It contains:

- Managed Audio EQ enable or bypass;
- Simple 3-band and Advanced 8-band editing;
- Flat, Voice Focus, Warm, Bass Reduce, and Treble Soften presets;
- combined headroom status and clipping protection;
- Adaptive Mix enable or bypass;
- strategy selection: Lecture Focus, Media First, Balanced, or Custom;
- a compact table of active apps with content type and priority controls;
- an explanation that only managed audio is affected.

The existing per-app inspector remains available from mixer rows. It is refined
into focused EQ and Adaptive sections rather than duplicating the entire Sound
workspace.

## Audio Model

### Managed Audio EQ

Add a persisted `GlobalEqualizerSettings` value with the same safe gain bounds,
band catalogs, presets, smoothing, and bypass semantics as per-app EQ.

Each managed stream uses this processing order:

1. per-app equalizer;
2. Managed Audio EQ;
3. combined headroom compensation;
4. manual per-app gain;
5. temporary adaptive gain;
6. output.

Headroom compensation accounts for both EQ layers so stacked boosts cannot clip
the callback buffer. Bypass must be bit-for-bit neutral when no other processing
layer changes the samples.

The backend stores the current global settings and applies them to existing and
newly created tap controllers. Excluded and unsupported apps retain saved
settings but are not described as actively processed.

### Adaptive app policy

Replace the user-facing adaptive role with two independent properties.

Content type:

- Lecture or Voice
- Meeting
- Music
- Video or Media
- Game
- Other

Priority:

- Foreground
- Normal
- Background
- Never Adjust

`Never Adjust` excludes an app only from adaptive gain. Manual volume and EQ
remain available.

Existing settings migrate as follows:

- Voice becomes Lecture or Voice with Normal priority.
- Media becomes Music or Video or Media based on the known app category, with
  Background priority.
- Ignore becomes Other with Never Adjust priority.
- Auto becomes the inferred content type with Normal priority.

The persisted decoder accepts existing 1.1.x values and writes the new policy on
the next successful save.

### Adaptive behavior

The background adaptive coordinator continues independently of window
visibility and evaluates live analysis at the existing low-frequency cadence.

Rules:

- The highest active priority establishes the current focus tier.
- Lower-priority apps are attenuated, never paused or muted.
- Foreground over Normal targets a gentle reduction.
- Foreground over Background targets a stronger reduction.
- Normal over Background targets a moderate reduction.
- Never Adjust always receives zero adaptive gain.
- Lecture or Voice and Meeting apps use speech activation and hang timing.
- Music, Video or Media, and Game apps use audibility rather than speech
  detection to establish activity.
- A background meeting cannot duck foreground media.
- Background music stays audible while foreground lecture speech remains clear.
- Attack and release stay smoothed so gains never snap or pump.
- Loudness correction remains bounded and cannot override the priority rules.

Strategies are curated starting points:

- **Lecture Focus** prioritizes Lecture or Voice apps and places Music in the
  background.
- **Media First** prioritizes Video or Media and Music while placing Meeting in
  the background.
- **Balanced** uses inferred content types with Normal priority.
- **Custom** preserves the user's explicit app policies.

Applying a strategy presents the resulting app policies before committing
changes when it would overwrite explicit custom choices.

### Sidechain Focus

Adaptive Mix exposes three focus-source modes:

- **Assigned Priorities** uses the persisted Foreground, Normal, Background, and
  Never Adjust policies directly.
- **Follow Front App** temporarily treats the audible frontmost managed app as
  Foreground.
- **Smart Hybrid** temporarily promotes an audible frontmost managed app by one
  priority tier while preserving explicit user priorities as guardrails.

Smart Hybrid is the recommended default. A Background app can rise only to
Normal, so a Background meeting cannot leapfrog Foreground media. A Normal app
can rise to Foreground. Never Adjust is never promoted and remains immune to
adaptive gain.

Frontmost status alone never triggers ducking. Lecture, voice, and meeting apps
must contain detected speech; music, video, media, game, and other apps must be
audible above the activity threshold. Attack and release remain smoothed, and
Sidechain Focus attenuates lower tiers rather than pausing or muting them.

## Onboarding and Repair

### First-run flow

First run becomes a progressive four-stage flow:

1. **Welcome and privacy** explains local processing and requires an explicit
   Continue before starting capture-capable work.
2. **Audio readiness** checks managed-audio support, capture authorization,
   output-device visibility, and route health with direct fix actions.
3. **Personalize Waves** offers common settings: appearance, theme, menu-bar
   visibility, launch at login, recent apps, shortcuts, and adaptive strategy.
4. **Ready** summarizes active settings, detected output, and the next useful
   actions.

The flow never silently changes launch-at-login, Accessibility, or other
permission-sensitive settings.

### Setup and Repair center

Settings gains a reusable **Setup & Repair** surface. The same surface can be
opened from main-window diagnostics and Help.

Each issue provides an action matched to its failure:

- Accessibility: open Privacy & Security > Accessibility, explain that Waves
  must be enabled, and re-check when the app becomes active.
- Audio capture: open the relevant Privacy & Security pane and re-check.
- Login item approval: open Login Items through `SMAppService`.
- Missing output: open Sound settings and refresh device state.
- Unhealthy routes: run route recovery and show the result.
- Unsupported macOS: explain the minimum version without offering a false fix.

**Redo Setup** is non-destructive. It reopens the guided flow with current
preferences and consent intact. A separate **Reset Waves** action, outside the
repair flow, requires confirmation and is not part of routine recovery.

System Settings URL construction lives in one service rather than inside a
single SwiftUI view.

## Data and Compatibility

New persisted values include:

- appearance selection;
- theme selection;
- Managed Audio EQ settings;
- adaptive strategy;
- per-app content type;
- per-app priority.

All new fields decode with safe defaults. Existing profiles, app intents,
per-device volumes, per-app EQ curves, pins, exclusions, and setup consent remain
intact. Schema changes must be additive or include explicit backward decoding.

Profiles do not store adaptive policy or Managed Audio EQ in 1.2.0. Profile
application remains limited to fields that are visible in the profile editor.

## Verification

### Automated

- strict Swift formatting and typechecking;
- all existing tests;
- global EQ coefficient, smoothing, bypass, stacked-headroom, and format tests;
- adaptive migration and policy tests;
- priority matrix, speech trigger, activity trigger, attack, release, and
  Never Adjust tests;
- persistence round trips and malformed-data tests;
- onboarding state, redo flow, and repair-action routing tests;
- appearance and theme persistence tests;
- release-script and appcast validation.

### Rendered macOS QA

- Waves and Graphite palettes in System, Light, and Dark appearances;
- main mixer with zero, one, two, and many apps;
- selected, hovered, excluded, managed, degraded, and error rows;
- per-app inspector and Sound workspace at minimum and default window sizes;
- first run and repair flows for every status;
- Increase Contrast, Reduce Transparency, and Reduce Motion;
- keyboard navigation and VoiceOver labels;
- menu-bar panel and Settings consistency.

### Audio behavior

Use deterministic offline streams and the live backend where available to prove:

- lecture speech over background music;
- media foreground over a background meeting;
- balanced music and meeting behavior;
- Never Adjust immunity;
- per-app plus Managed Audio EQ composition;
- device changes and route recovery while adaptive mixing is enabled;
- app startup after policies were saved while the app was not running.

Any hardware, permission, or provider limit that prevents a live audible check is
reported separately and does not get replaced by a narrower claim.

## Release

The release target is **Waves 1.2.0, build 4**.

Release order:

1. reconcile and commit the current 1.1.1 work with the 1.2.0 feature work;
2. finish changelog, cask, version, build, and release-note metadata;
3. run the full local build, test, package, and isolated-runtime gates;
4. sign with the existing Developer ID identity;
5. notarize with the existing Keychain profile without exposing credentials;
6. staple and validate the DMG and contained app;
7. run publication checks and package smoke tests;
8. generate and validate the Sparkle appcast;
9. publish the GitHub release asset and aligned tag;
10. publish the appcast and download metadata to the Waves site;
11. update the Homebrew cask metadata;
12. verify that the installed 1.1.0 app discovers and installs 1.2.0 through
    Sparkle, then verify the installed bundle and live UI.

No production release is published unless every required gate passes against the
same commit and same artifact set.
